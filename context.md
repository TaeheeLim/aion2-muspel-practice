# 무스펠의 성배 신호등 연습기 — 3D 버전 컨텍스트

이 문서는 현재 **2D 캔버스 웹게임**(`index.html`)을 **3D 버전**으로 다시 만들 때 필요한 모든 맥락을 담는다.
새 세션/개발자가 이 문서만 읽고 바로 착수할 수 있도록 게임 규칙을 빠짐없이 명세한다.

---

## 1. 프로젝트 개요

- **무엇**: 아이온2 신규 성역 **'무스펠의 성배'**(보스: 지저의 재앙 칼드릭스)의 "신호등(구슬 색맞추기)" 기믹 **연습기**.
- **목표**: 보스로 다가오는 구슬의 색을, 바닥에 깔리는 **장판**으로 바꿔서 보스 주위 **3겹 테두리 색**과 맞춰 넣는 패턴을 반복 연습.
- **현재 상태**:
  - 2D 구현: [`index.html`](index.html) (단일 파일, 의존성 없음, HTML5 Canvas 2D)
  - 문서: [`README.md`](README.md)
  - 배포: GitHub Pages — https://taeheelim.github.io/aion2-muspel-practice/
  - 저장소: https://github.com/TaeheeLim/aion2-muspel-practice (Public)
- **3D로 만들려는 이유**: 평면 탑다운이 아니라 실제 인게임처럼 입체 공간에서 보스/구슬/장판/캐릭터를 보며 연습하고 싶음.

## 2. 게임 규칙 (확정 명세 — 2D에서 검증 완료)

> 아래는 사용자가 실제 인게임 기준으로 확정해 준 규칙이다. 3D에서도 동일하게 동작해야 한다.

### 2.1 보스와 테두리(링)
- 중앙에 보스. 보스 주위에 **원형 테두리 3겹**(바깥→중간→안쪽).
- 3겹 색은 매 라운드 **빨/초/파의 무작위 순열** — **반드시 서로 다른 3색**(중복 불가, 예: 초·파·파 불가능).
- 테두리(바깥→안쪽)는 각각 **1번·2번·3번 구슬의 목표 색**이다.

### 2.2 구슬 레인
- 보스 중심의 **시계 위치**(12,1,2,...,11시)에서 구슬이 **한 줄에 3개**씩 보스로 천천히 이동.
- **1·4·7·10시**는 **두 줄(2열 × 3개 = 6구슬)**로 나온다. 두 줄은 같은 진입 스케줄을 공유.
- 각 구슬의 초기 색은 **무작위(빨/초/파)**. 단,
  - 한 줄의 구슬 3개가 **모두 같은 색일 수는 없다(최소 2색)**.
  - 등장 색 조합은 **반드시 폭발 스케줄 안에서 테두리 색으로 맞출 수 있는 조합만** 나온다(불가능하면 자동 재추첨 — solvability 보장).

### 2.3 구슬 ↔ 테두리 매칭
- 한 줄의 **맨 앞(보스에 가까운) 구슬 = 가장 바깥 테두리 색**, 2번째 = 중간, 3번째 = 안쪽 테두리 색과 같아야 **정답**.
- (구슬은 줄 안에서 앞→뒤 = 테두리 바깥→안쪽 순서로 대응. 코드상 `tr`(0/1/2) = 목표 링 인덱스.)

### 2.4 장판(빔)으로 색 변경
- 장판은 **캐릭터를 따라다니며** 생성되어, **보스 → 캐릭터 → 바깥 끝**까지 이어지는 **직선**이다. 길이는 보스와의 거리와 **무관**(항상 가장자리까지).
- 장판은 시간이 지나며 **점점 진해지고**, **거의 다 차오르면(약 85%) 캐릭터에서 분리·고정**된 뒤 폭발한다.
- **폭발하면 그 직선이 지나는 '한 줄' 전체(최대 3개)**의 색이 **빨→초→파→빨→초…** 순서로 한 칸 바뀐다.
- **한 번에 무조건 1줄만** 변경된다(두 줄 레인이면 둘 중 더 가까운 한 줄만). 바꾸기 싫은 줄은 **조준을 피하면** 된다.
- 조준 보정: 장판은 가까운 줄에 **자석처럼 정렬(snap)**되어, 살짝 어긋나도 그 줄 3개 모두 확실히 바뀐다.

### 2.5 진입 타이밍(폭발 횟수 기반)
- 구슬은 시간/거리가 아니라 **장판 폭발 누적 횟수**로 보스에 들어간다.
- **3번 터지면 1번째 구슬**, 이후 **2번 더(누적 5번)에 2번째**, 다시 **2번 더(누적 7번)에 3번째**가 진입.
- 단, 임계 폭발에 도달하면 **색이 확정·고정**되고, **약 1초 후**에 실제로 보스로 빨려 들어간다(진입 연출).

### 2.6 진입 결과
- 구슬이 들어갈 때, **그 구슬 색과 같은 색의 테두리가 한 겹씩 사라진다**(정답이든, 게임오버 전 1개째 오답이든).
- **잘못된 색 구슬이 2개 들어가면 게임 실패**(게임오버).

### 2.7 조작
- 캐릭터를 **이동만** 하면 된다(장판은 자동 생성). 키보드 `WASD`/방향키, 또는 마우스로 끌기.
- 시작 시 캐릭터는 **사용자가 선택한 시작 위치(시계)**에 배치된다.

## 3. 현재 2D 구현의 핵심 수치 (튜닝 기준값)

`index.html` 상단 상수 — 3D에서도 그대로 출발점으로 쓰면 된다.

| 상수 | 값 | 의미 |
|---|---|---|
| `COMMIT` | `[3,5,7]` | 1·2·3번 구슬이 진입하는 누적 폭발 횟수 |
| `ENTER_DELAY` | `1.0`초 | 색 확정 후 실제 진입까지 지연 |
| `ARM_TIME` | `3.0`초 | 장판 생성→폭발 주기 = **게임 전체 속도**(키우면 느려짐) |
| `LOCK_FRAC` | `0.85` | 장판이 캐릭터에서 분리·고정되는 충전 비율 |
| `BEAM_TOL` | `0.22` rad | 장판이 줄에 자석 정렬되는 허용각 |
| `COL_OFFSET` | `0.16` rad | 두 줄(열) 사이 각도(약 9°) |
| `DOUBLE_CLOCKS` | `{1,4,7,10}` | 두 줄(6구슬)로 나오는 시계 위치 |
| 색 인덱스 | `0=빨(#ff4d5e) 1=초(#27d97a) 2=파(#4db5ff)` | 순환 순서 빨→초→파 |
| 이동 속도 | `arenaR*0.8`/초 | 캐릭터 키보드 이동 |

핵심 로직 함수(2D): `spawnLane()`, `laneSolvable()`(완전탐색 solvability 검사), `explode()`(한 줄 색 변경+테두리 제거 트리거), `commitOrb()`(진입 판정+테두리 제거+실패), `orbRad()`(폭발 진척에 따른 구슬 위치).

## 4. 2D → 3D 매핑

| 2D 요소 | 3D 표현(권장) |
|---|---|
| 탑다운 원형 아레나 | 원형 바닥 평면(원형 플랫폼) + 약간 기울인 3인칭/쿼터뷰 카메라(또는 인게임형 등각) |
| 보스 = 원 + 키아트 이미지 | 보스 메시(용/벌레형). 임시로는 실린더/구체 + 텍스처. 이상적으로 칼드릭스 풍 모델 |
| 3겹 테두리(원) | 보스 둘레의 **3겹 발광 링(torus)** — 색 입히고, 제거 시 페이드/축소 연출 |
| 시계 위치 레인 | 보스 중심에서 방사형으로 뻗는 12방향. 구슬은 해당 방향 라인을 따라 보스로 이동 |
| 구슬(작은 원) | 발광 구체(emissive sphere), 색 = 빨/초/파. 위에 1/2/3 번호 라벨(빌보드) |
| 장판(방사형 직선 사각형) | 바닥에 깔리는 **발광 빔/데칼**(보스→캐릭터→가장자리). 진해지는 emissive, 폭발 시 충격파 |
| 캐릭터(점) | 작은 3D 캐릭터/마커. 이동은 바닥 평면 위 XZ 이동 |
| 좌상단 범례 / 폭발 N/7 HUD | DOM/HTML 오버레이(2D UI) 또는 빌보드 텍스처 |
| 진입 연출(보스로 빨려듦) | 구슬이 보스 쪽으로 이동하며 스케일↓ + 흡수 파티클 |
| 테두리 제거 | 해당 링 메시 페이드아웃 + 링 파편/플래시 |

각도 규칙(2D 그대로 사용 가능): 시계 h시 방향 단위벡터 `dir = (sin(h·30°), -cos(h·30°))`. 3D 바닥(XZ평면)이면 `x=sin, z=-cos`(또는 화면 좌표계에 맞춰 y/z 매핑).

## 5. 추천 기술 스택

### 기본 추천: **Three.js (웹 3D)**
- 이유: 현재 프로젝트가 웹 단일 배포(GitHub Pages)라 **그대로 같은 URL/배포 파이프라인** 유지 가능. 설치 없이 브라우저 실행. 게임 규모(보스1+구슬 최대 6+링3+빔)가 가벼워 충분.
- 구성: `three` (+ 선택적으로 `OrbitControls`, 후처리 `EffectComposer`로 bloom). 단일 `index3d.html` 또는 Vite + 모듈.
- 입력/HUD: 기존 2D UI 패턴(DOM 오버레이) 재사용.
- 에셋: 보스는 `GLTFLoader`로 모델 로드(없으면 프리미티브+텍스처로 대체). 현재 키아트(`assets/cand2.jpg`)는 배경/임시 텍스처로 활용 가능.

### 대안
- **react-three-fiber (R3F)**: Three.js를 React로. 상태관리·컴포넌트화가 편하지만 빌드 필요.
- **Babylon.js**: 웹 3D 풀기능 엔진. 물리/파티클 내장.
- **Unity / Godot**: 본격 게임/콘솔·앱 배포를 원하면. 웹 즉시배포·가벼움 면에선 Three.js가 유리. 무스펠 실제 모델/이펙트를 제대로 만들 거면 Unity가 에셋 파이프라인이 강함.

> 별다른 이유가 없으면 **Three.js 단일 페이지**로 시작하고, 규칙 로직은 2D 버전의 순수 함수(스폰/솔버/스케줄)를 거의 그대로 포팅하는 것을 권장.

## 6. 3D 구현 단계별 계획

1. **씬 골격**: 원형 바닥 + 카메라(쿼터뷰) + 라이트 + 보스 자리(임시 메시) + 3겹 토러스 링.
2. **레인/구슬**: 시계 방향 라인, 구슬 구체 스폰(색/번호), 폭발 진척(`orbRad`)에 따른 보스 방향 이동.
3. **캐릭터 이동 + 장판**: 바닥 XZ 이동, 캐릭터 추종 빔(보스→캐릭터→가장자리), 진해짐→85% 고정→폭발.
4. **규칙 로직 포팅**: `COMMIT/ENTER_DELAY/ARM_TIME/LOCK_FRAC/BEAM_TOL`, 한 줄만 색변경, 줄 자석 정렬, 진입 스케줄, 1초 진입 지연, 테두리 제거, 2회 실패, `laneSolvable` 재추첨.
5. **HUD/UX**: 시작 위치 선택, `폭발 N/7`, 테두리 범례(또는 3D 링 자체로 대체), 실패 오버레이.
6. **연출/폴리시**: bloom, 폭발 충격파, 진입 흡수 파티클, 사운드, 모바일 입력(드래그).
7. **에셋 고도화**: 보스 모델, 던전풍 배경. (저작권 주의 — 7절 참고)
8. **배포**: 같은 저장소에 `3d/` 또는 `index3d.html` 추가 → Pages 경로로 노출.
